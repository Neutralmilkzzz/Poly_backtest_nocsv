# R 回测平台说明书

## 1. 这套目录是干什么的

这个项目现在还没有开始写策略，它目前处在“搭建回测平台骨架”的阶段。

回测平台可以先理解成一条流水线：

1. 读取原始市场数据
2. 清洗并整理成适合回测的数据格式
3. 把策略接进来，生成买卖信号
4. 按成交规则模拟下单和持仓变化
5. 计算收益、回撤、胜率等指标
6. 输出图表、日志和结果文件

所以，目录结构本质上是在为这条流水线预留位置。

## 2. 你现在的目录分别代表什么

### 根目录

根目录尽量保持干净，不直接堆满数据和脚本。根目录的作用主要是：

1. 放项目主入口文件
2. 放最核心的说明文档
3. 放项目级配置

你现在看到的大部分内容已经被分流到不同文件夹，这是正确方向。

### data/

这是数据层，专门管理市场数据。

#### data/raw/

这里放原始数据，不做人工修改。

你的 Polymarket 比特币 5 分钟市场 CSV，应该长期保存在这里。

这层的原则是：

1. 原样保存
2. 不混入分析结果
3. 后续任何清洗都从这里读取

#### data/processed/

这里放处理后的数据。

比如未来你可能会把原始逐笔盘口数据处理成：

1. 每个市场一张标准化表
2. 某个统一时间步长的数据
3. 回测直接使用的特征表

这样做的好处是，回测时不必每次都重新清洗原始 CSV。

### R/

这是 R 语言主代码目录，通常也是整个项目最核心的目录。

它下面已经分了几个模块：

#### R/io/

这里负责“输入输出”。

未来通常会放：

1. 读取 CSV 的函数
2. 读取配置文件的函数
3. 保存结果表格的函数
4. 保存图表和日志的函数

简单理解，这里负责“把东西读进来”和“把东西写出去”。

#### R/engine/

这里是回测引擎层。

这是整个平台最关键的部分，未来通常负责：

1. 按时间推进市场状态
2. 接收策略信号
3. 模拟成交
4. 更新仓位、现金、净值
5. 记录每一笔交易和账户变化

如果把整个系统比作汽车，这里就是发动机。

#### R/metrics/

这里负责绩效评估。

未来通常会计算：

1. 累计收益
2. 年化收益
3. 最大回撤
4. 胜率
5. 盈亏比
6. Sharpe Ratio
7. 每笔交易统计

回测不是只看赚没赚钱，更要看赚得稳不稳，这一层就是干这个的。

#### R/utils/

这里放通用工具函数。

比如：

1. 时间处理
2. 参数检查
3. 数据字段校验
4. 通用格式转换
5. 小型辅助函数

这层不直接做策略，也不直接做撮合，而是给别的模块打下手。

### strategies/

这里专门放策略。

你已经说了“策略已经有了，但那是下一步”，所以这个目录是对的。将来每个策略尽量独立放置，不要把策略逻辑写死在引擎里。

推荐原则：

1. 策略只负责发信号
2. 引擎只负责执行和记账
3. 不要把成交逻辑写进策略文件

这样以后换策略时，不用重写整个平台。

### scripts/

这里放可执行脚本。

它和 R/ 的区别是：

1. R/ 里放可复用函数
2. scripts/ 里放“运行某件事”的脚本

例如未来可能会有：

1. 清洗原始数据脚本
2. 运行单次回测脚本
3. 批量参数测试脚本
4. 生成报告脚本

可以把 scripts/ 理解为“按钮层”。

### config/

这里放配置文件。

未来建议把这些内容从代码里剥离出来：

1. 数据路径
2. 初始资金
3. 手续费
4. 滑点设置
5. 回测时间范围
6. 策略参数

这样以后调参数时不需要直接改 R 代码。

### results/

这里放结构化结果文件。

比如未来可能保存：

1. 每次回测的净值曲线数据
2. 交易明细
3. 仓位记录
4. 参数组合结果

这是“机器可读结果”。

### reports/

这里放给人看的结果输出。

比如：

1. Markdown 报告
2. HTML 报告
3. 图表快照
4. 回测总结文档

这是“人类可读结果”。

### logs/

这里放运行日志。

未来在回测出错或结果异常时，日志非常重要。常见用途：

1. 记录回测开始和结束时间
2. 记录加载了哪些数据
3. 记录参数值
4. 记录警告和报错

### tests/

这里放测试代码。

当前有一个 testthat/，说明这套结构已经按 R 常见测试习惯预留好了位置。

未来可以测试：

1. 数据读取是否正确
2. 撮合逻辑是否正确
3. 手续费计算是否正确
4. 净值更新是否正确
5. 策略信号是否符合预期

## 3. 一个专业回测平台是怎么分层的

对于新手来说，最容易混乱的地方是：什么属于策略，什么属于引擎，什么属于数据处理。

你可以这样记：

### 第一层：数据层

回答的问题是：

“市场原始数据长什么样？”

这一层负责读取、清洗、标准化数据。

### 第二层：策略层

回答的问题是：

“在这个时刻，我想买还是卖，还是不动？”

这一层只负责产生信号，不负责真正成交。

### 第三层：执行层

回答的问题是：

“假设策略发了买入信号，按什么价格成交？成交多少？要不要扣手续费？”

这就是回测引擎的核心。

### 第四层：账户层

回答的问题是：

“现在持仓多少？现金多少？总资产多少？”

这一层会更新账户状态。

### 第五层：评估层

回答的问题是：

“这套策略到底好不好？”

这一层负责输出绩效指标和图表。

## 4. 用你的 Polymarket 数据，未来大概会怎么流动

你现在的数据是 Polymarket 比特币 5 分钟大小市场的逐步更新数据。未来很可能会走下面这条路径：

1. 从 data/raw/ 读取原始 CSV
2. 在 R/io/ 中完成读取和字段标准化
3. 在 data/processed/ 中保存整理好的回测输入表
4. 策略从 processed 数据中读取状态，生成信号
5. 引擎根据盘口或成交价规则做模拟成交
6. 结果保存到 results/
7. 总结报告输出到 reports/

这就是整个平台的主工作流。

## 5. 回测新手最容易踩的坑

### 1. 把策略和引擎写在一起

这样短期看起来快，长期会很难维护。

正确思路是：

1. 策略只说“我想做什么”
2. 引擎负责“系统实际上怎么做”

### 2. 直接拿 midpoint 当作真实成交价

midpoint 可以用于研究，但不一定代表真实可成交价格。

对于盘口市场，真实回测通常更接近：

1. 买入按 ask
2. 卖出按 bid

否则结果会偏乐观。

### 3. 忽略手续费和滑点

很多策略“看起来赚钱”，是因为没扣交易成本。

### 4. 原始数据被反复修改

原始数据一旦被覆盖，后面很难回溯问题。

所以 raw/ 一定尽量只读不改。

### 5. 没有测试就直接信回测结果

如果撮合、持仓更新或手续费计算错了，整份回测都可能失真。

## 6. 你可以怎么理解“现在还缺什么”

你现在已经有了“房子的结构”，但还没有“家具和电路”。

具体来说，接下来真正要逐步补的是：

1. 数据读取函数
2. 数据清洗函数
3. 回测引擎主流程
4. 成交规则
5. 账户更新规则
6. 绩效统计函数
7. 策略接入接口
8. 运行脚本

现在没有开发这些内容是正常的，因为你当前目标只是先把平台骨架搭好并理解它。

## 7. 一个最实用的理解方式

你可以把整个项目看成三句话：

1. data/ 负责存数据
2. R/ 负责写能力
3. scripts/ 负责运行流程

再补两句：

1. strategies/ 负责表达交易想法
2. results/、reports/、logs/ 负责保存输出

如果你先把这五句话记住，后面看代码就不容易乱。

## 8. 你下一步最合理的学习顺序

如果你是回测新手，建议按这个顺序学，而不是一上来就写完整策略：

1. 先学会区分 raw 数据和 processed 数据
2. 再学会定义一笔交易是怎么成交的
3. 再学会账户净值是怎么更新的
4. 再把策略信号接进引擎
5. 最后再做绩效统计和报告

这个顺序比“直接上策略”更稳。

## 9. 一句话总结

你现在这个项目架构不是在“做策略”，而是在“为策略准备一个能长期复用的回测平台”。

这是专业做法，因为以后你可以不断替换策略，但不必反复重写数据层、引擎层和评估层。
